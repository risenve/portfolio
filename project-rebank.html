Product Design · Design System

ReBank DS

Token-based design system —
from Figma to production code

NDA notice: The original product is under NDA. This case reflects the real architecture, process, and design decisions — product name and visuals have been adapted for public sharing.

Role
Product Designer (solo)
Stack
Figma · Figma Tokens · Style Dictionary v5 · Storybook 10 · React · CSS
Scope
Design system + 5 screen flows · Light & Dark themes

Most design systems stop at Figma. This one doesn't.

ReBank DS is a token-based design system built for a mobile banking product — from raw color values in JSON all the way to documented React components in Storybook. Every visual decision in the product UI traces back to a token. Every token compiles into a CSS variable that lives in code.

The goal was to close the gap between design and development: one source of truth, no manual handoff of color values, no "which blue is this?" conversations.

01

Token Architecture

Three-layer token model

Tokens are structured in three layers. This is not cosmetic — it determines how the system scales and how theming works.

Primitives   →   raw values, no meaning attached
             color.blue.500 = #1A6BFF
             spacing.4     = 16px

Semantic     →   intent-based aliases
             color.background.primary = {color.blue.500}
             color.text.danger        = {color.red.400}

Component    →   scoped overrides per component
             button.background.default = {color.background.primary}

When the brand color changes, you update one primitive. Every semantic token that references it updates automatically across the entire product.

Themes

$themes.json defines Light and Dark as separate theme configurations, both referencing the same semantic layer. Switching themes doesn't require changing component code — only the active theme variable changes.

02

Build Pipeline

Tokens are not static — they compile.

Style Dictionary reads JSON sources from Primitives/, Semantic/, and Typography/, and outputs CSS custom properties and JS variables into build/. A watch script rebuilds on every file change.

Figma Tokens (JSON)
      ↓
Style Dictionary config
      ↓
build/
  ├── variables.css    ← CSS custom properties
  └── tokens.js        ← JS object for runtime use

This means the design token file IS the codebase. There's no intermediate step where a developer copies values from a Figma spec. The CSS is generated.

03

Storybook

Components are documented in Storybook using MDX stories. The setup uses @storybook/react-vite and @storybook/addon-docs for rich documentation pages — not just a component sandbox, but a living spec.

Each story imports CSS variables from build/variables.css, so what renders in Storybook uses the same token values as the product.

04

Component Library in Figma

In Figma, components are organized by page with usage rules: when to use, when not to use, spacing behavior, state variations, and token references. This documentation lives alongside the components themselves, not in a separate spec file.

05

Product Screens

The system was validated by building five real screen flows on top of it. Every color, spacing value, and typography style in these screens comes from a token — nothing is hardcoded.

Auth

Sign In & Registration

Standard entry point. Two states: returning user (sign in) and new user (registration). Phone number input, OTP confirmation.

Onboarding

Product Walkthrough

Guided introduction to product features before the user commits to registration. 3–4 slides with illustration and CTA.

KYC

Identity Verification

The most complex flow in the product. Multi-step verification requiring document scan and face recognition — common in regulated fintech products, rare in design portfolios.

Document Verification

User photographs their passport or ID card. The flow handles: instructions screen → active scanning → processing → success/error states.

Face Scan

Liveness check via front camera. Five distinct states designed: idle, aligning, scanning (oval active), failure, success.

Verification Confirmation

Three-step confirmation after document and face check pass. Transition to account activation.

Add

Account Setup

Post-verification steps: adding a bank card or device to the account.

Home

Banking Dashboard

The main product screen. Two theme variants (Light / Dark) + detailed widget view. This screen shows the full density and maturity of the system.

06

Design Decisions

Why not use an existing design system — MUI, Tailwind?

Banking products have strict visual requirements: specific color contrast ratios for accessibility, precise touch targets, state coverage for every interactive element. A custom token-based system gives full control without fighting against someone else's opinions.

Why Style Dictionary over manual CSS?

Tokens edited in JSON and compiled automatically are harder to break than tokens copied manually into a stylesheet. The pipeline makes consistency enforced, not voluntary.

Why document in both Figma and Storybook?

Designers work in Figma. Developers work in code. Storybook bridges them — a developer can see a component rendered with real tokens, with documented props, without opening Figma. This reduces the "what did the designer mean?" loop.

Retrospective

What I'd do differently

  • Set up Chromatic for visual regression testing — right now there's no automated way to catch if a token change breaks a component visually.
  • Add a changelog to the token files so the team can track what changed between builds.
  • Deploy Storybook to a public URL (GitHub Pages or Chromatic) so it's shareable without cloning the repo.
